## ITENS PENDENTES (5)

### 1. Desligar BuildMainMenu runtime
UIController.BuildMainMenu() (linha 565-594) instancia em runtime um menu paralelo que vai sobrepor o canvas estático. Precisa parar de criar o `mainMenuPanel`.

Restrições:
- O resto do UIController (ShowScreen, OnEnterDungeonClicked, BeginDungeonRun, ReturnToMainMenu, RefreshMainMenu) é referenciado em ~6 lugares. Não posso quebrar.
- ShowScreen(Screen.MainMenu) usa `mainMenuPanel.gameObject.SetActive(s == Screen.MainMenu)` — se mainMenuPanel for null, NRE.
- RefreshMainMenu() escreve em `mainMenuStatus.text` e `menuEnterDungeonBtn.interactable` — se forem null, NRE.

Opção A: Comentar/remover BuildMainMenu inteiro + as chamadas a Refresh/SetActive em ShowScreen para o caso MainMenu. Cena é a fonte da verdade.
Opção B: Esvaziar BuildMainMenu (deixa só as referências necessárias setadas como null safe), mas RefreshMainMenu e ShowScreen ainda tentam usá-las.
Opção C: Cachear referências do canvas estático em fields e fazer RefreshMainMenu atualizar o canvas (chip values, dive hint), e ShowScreen ativar/desativar MainMenuCanvas.

Recomendação técnica desejada: qual minimiza risco e mantém consistência com Screen system?

### 2. Wirar onClick do DIVE e PARTY
- DIVE button (DiveButton/Button) → equivalente do antigo `OnEnterDungeonClicked` (RefreshRoster + ShowScreen(Screen.Roster))
- NavParty button → mesmo destino (manage party screen)

Como UIController constrói menus em runtime no Awake/Start, o `Button.onClick` do canvas estático precisa ser wirado de algum jeito.

Opção A: Wirar via Inspector usando UnityEvent persistente (drag-drop, mas eu uso MCP — preciso definir o `m_PersistentCalls` do Button via MCP).
Opção B: Adicionar `[SerializeField]` no UIController para os 2 buttons, plus um pequeno `Awake` que faz `.AddListener` (single line por button, NÃO é "build menu", é só wiring).
Opção C: Criar um component novo `MainMenuBindings` que tem refs para os buttons + chama UIController via SendMessage/UnityEvent.

Qual é mais correto/idiomático para Unity, considerando que NÃO posso construir menus em runtime?

### 3. Pulse animation nos rings
PulseRing1 e PulseRing2 são Image circulares estáticos. CSS original tinha:
- scale 0.85 → 1.25 + opacity 0.7 → 0 ao longo de 2.4s, loop infinito
- Ring2 com delay 1.2s (offset metade do ciclo)

Implementação Unity:
Opção A: Animator + AnimationClip com keyframes em scale e Image.color.alpha. Loop=true. Estado único, animação rolada do início.
Opção B: DOTween (não está no projeto, addiria dependência).
Opção C: Coroutine em Start (mas isso é "runtime build" e foi proibido).
Opção D: Uma classe pequena `UIPulse` MonoBehaviour reutilizável (Update com Mathf.Repeat) — ainda é código mas é declarativo, não constrói hierarchy.

Qual a abordagem mais limpa para Mockup B mobile-RPG, sem adicionar deps?

### 4. Atmosphere blobs — softness
Hoje são Image retangulares com cor flat. Em mobile mockup B, deveriam ser blobs radiais soft (CSS `radial-gradient(circle, color 0%, transparent 25%)`).

Já existe `Assets/UI/Sprites/soft_blob.png` (radial alpha falloff smoothstep).

Plano: trocar o `Image.sprite` dos 3 atmosphere objects para soft_blob.png e ajustar `color.a` para que o blob mantenha intensidade visual (0.12 / 0.10 / 0.18 → testar 0.4 / 0.35 / 0.5 com sprite soft).

Pergunta: blend mode? Image padrão usa SrcAlpha/OneMinusSrcAlpha. Para "luz aditiva" (atmosphere), seria melhor usar Material com Additive blend? Trade-off: 1 material novo vs visual fidelity vs dark theme.

### 5. PLAY FAB — outline polish
PlayFab (gold circle 56x56) usa Outline (UnityEngine.UI.Outline) com cor #0e0a1f e effectDistance 3,-3 — simula a "border 3px solid #0e0a1f" do CSS, mas em retângulo (Outline funciona com qualquer Image; com sprite circular ele simplesmente desloca cópias do sprite, criando halo).

Resultado: pode ficar com "borda" assimétrica ou borrada, não um anel limpo.

Alternativas:
Opção A: Remover Outline. Em vez disso, criar um child Image atrás do FAB (PlayFab_BorderRing): mesmo sprite circle_solid.png, cor #0e0a1f, sizeDelta 62x62 (3px maior em cada lado). Render order garante que apareça atrás (parent ordering ou siblingIndex).
Opção B: Manter Outline, ajustar effectDistance para 2/2 (mais sutil).
Opção C: Criar um shader custom para anel circular limpo (overkill).

Qual é mais idiomática para Unity uGUI sem custom shader?
