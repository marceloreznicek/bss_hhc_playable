## CONTEXTO ORIGINAL DO PRECODE

# PRÉ-CODE — HeroMerge v0.3 (4 fixes na mecânica de grid-merge)

## TAREFA

A v0.2 do HeroMerge tem um grid 4×2 onde o player tapeia SUMMON pra
adicionar Tier-1 e arrasta uma célula em cima de outra de mesmo tier
pra mesclar (2-of-a-kind → tier+1). O player pediu **4 mudanças** pra
v0.3:

1. **3-of-a-kind**: merge agora exige 3 unidades do mesmo tier, não 2.
2. **Adjacência obrigatória**: a unidade arrastada precisa ser solta em
   cima de outra que esteja **lado a lado** com uma terceira de mesmo
   tier — em outras palavras, o cluster final precisa ter pelo menos 2
   das 3 células geometricamente adjacentes (compartilhando borda
   horizontal ou vertical no grid 4×2).
3. **Pulsing magnetic pull**: enquanto arrasta, mostrar visualmente
   QUAIS 3 células serão consumidas. Não apenas highlight — animação de
   "pull" pulsante (escala/glow/shake) nas células do cluster que vai
   ser mergeado, atualizada em tempo real conforme o dedo se move.
4. **Não overconsume**: se houver MAIS de 3 do mesmo tier disponíveis
   (ex: 4 Tier-1s no grid), o merge deve consumir EXATAMENTE 3 (a
   arrastada + as 2 melhores candidatas). Sobra 1 intacta no grid. Em 5,
   sobra 2. Etc.
5. **Drag offset bug**: hoje o ghost que segue o dedo aparece **deslocado
   pra esquerda** porque o `#drag-ghost` usa `position: absolute` dentro
   de `#bottom-panel` mas o CSS `transform: translate(-50%, -50%) scale(1.15)`
   roda DEPOIS do `left/top`, e o `top: 1000px` de `#bottom-panel` mais o
   reflow dá um offset visual. Resultado: o sprite arrastado aparece à
   esquerda do dedo, não centralizado. Precisa ficar exatamente embaixo do dedo.

Vocês são 8 IAs. Apontem:
- Edge cases na lógica de "escolher 2 vizinhas + a arrastada"
- Race conditions entre o pulse animation e o pointermove
- Bugs prováveis na seleção de "melhores candidatas" (qual critério? distância? índice? dist Manhattan adjacente?)
- Riscos no offset do ghost (transform-origin? CSS specificity? scale × translate ordering?)
- Qualquer pegadinha de UX (jogador vê 4 pulsando, escolhe arrastar, e só 3 mesclam — confunde?)
- Soluções alternativas

Restrição: ES5-ish JS, sem novas dependências, mudança mínima.

## PLANO DE IMPLEMENTAÇÃO PROPOSTO

### A. Resolver cluster de 3 (`computeMergeCluster(srcIdx, dropIdx)`) — ~50 LoC

```js
// Adjacency map for the 4×2 grid (precomputed)
var GRID_ADJ = [
  [1,4], [0,2,5], [1,3,6], [2,7],
  [0,5], [1,4,6], [2,5,7], [3,6]
];

// Returns up to 2 grid indices to PAIR with the dragged cell + dropTarget.
// Selection priority:
//   1. dropTarget itself (must exist + same tier + adjacent to ≥1 other same-tier)
//   2. one same-tier cell adjacent to dropTarget (closest by Manhattan)
// Returns array of 3 indices [srcIdx, dropIdx, partnerIdx] or null if no valid trio.
function computeMergeCluster(srcIdx, dropIdx) {
  var s = STATE.grid[srcIdx];
  if (!s) return null;
  if (s.tier >= 5) return null;
  var d = STATE.grid[dropIdx];
  if (!d || d.tier !== s.tier || dropIdx === srcIdx) return null;
  // Candidates = same-tier cells adjacent to dropIdx (excluding srcIdx)
  var adj = GRID_ADJ[dropIdx];
  var partner = -1;
  for (var i = 0; i < adj.length; i++) {
    var a = adj[i];
    if (a === srcIdx) continue;
    var u = STATE.grid[a];
    if (u && u.tier === s.tier) { partner = a; break; }
  }
  if (partner < 0) return null;
  return [srcIdx, dropIdx, partner];
}
```

### B. Modificar `tryMergeCells` — exatamente 3, próximo tier no dst — ~15 LoC

```js
function tryMergeCells(srcIdx, dstIdx) {

  if (!cluster) return false;

  STATE.grid[dstIdx] = { tier: nextTier };
  STATE.mergesCompleted++;

  spawnGridFlash(dstIdx);
  return true;
}
```

```js
function updatePullPreview(srcIdx, hoverIdx) {
  var cells = document.querySelectorAll('.grid-cell');
  for (var i = 0; i < cells.length; i++) cells[i].classList.remove('pulling');
  if (hoverIdx < 0 || hoverIdx === srcIdx) return;
  var cluster = computeMergeCluster(srcIdx, hoverIdx);
  if (!cluster) return;
  for (var k = 0; k < cluster.length; k++) {
    var c = document.querySelector('.grid-cell[data-idx="' + cluster[k] + '"]');
    if (c) c.classList.add('pulling');
  }
}
```

### D. Fix do drag-ghost offset — ~5 LoC CSS

Trocar `transform: translate(-50%, -50%) scale(1.15)` por usar `margin-left/top: -100px` (metade do width=200) que NÃO pega no compositor antes do `left/top` se aplicar. Ou mover `#drag-ghost` pra fora do `#bottom-panel` (anexar em `#game`) e usar coords no espaço do `#game` direto.

Solução escolhida: anexar ghost a `#game` (não a `#bottom-panel`) e calcular offset usando `getBoundingClientRect()` de `#game`. O scale do harness é uniforme, então `clientX - gameRect.left` dividido por `(gameRect.width / 1080)` dá game-space x. Setar `left/top` nesses coords. Mantém `translate(-50%, -50%)` mas agora dentro de um container que não tem padding/transforms que confundam.

OU mais simples: parar de usar `translate(-50%, -50%)` e em vez disso setar `ghost.style.left = clientX - 100 + 'px'` direto (no client space) usando `position: fixed` em vez de `absolute`. `position: fixed` ignora o pai e ancora no viewport — exatamente o que queremos pra ghost que segue o dedo no client space.

```css
#drag-ghost {
  position: fixed;
  display: none;
  width: 200px; height: 200px;
  pointer-events: none;
  z-index: 9999;
  filter: drop-shadow(0 0 18px rgba(255,210,80,0.85));
  /* removed: transform translate; left/top now centered manually */
}
```

```js
function moveDragGhost(clientX, clientY) {
  var ghost = document.getElementById('drag-ghost');
  ghost.style.left = (clientX - 100) + 'px';  // 100 = width/2
  ghost.style.top  = (clientY - 100) + 'px';
}
```

Agora ghost segue o dedo exatamente. `position: fixed` ancora no viewport, ignora qualquer transform de pais. Mais simples, mais robusto.

# Revisão Claude Opus 4.7 — HeroMerge v0.3

## 🔴 Bugs críticos na lógica de cluster

**1. `partner === dropIdx` impossível mas não validado.** Em `computeMergeCluster`, `GRID_ADJ[dropIdx]` nunca contém `dropIdx`, então OK — mas documente ou adicione `if (a === dropIdx) continue` defensivo.

**2. `partner === srcIdx` já é filtrado** (linha `if (a === srcIdx) continue`). ✅

**3. BUG REAL — `dropIdx === srcIdx` check vem DEPOIS do acesso a `d`.** Se `srcIdx === dropIdx`, `d = s` (mesma célula), passa no `d.tier !== s.tier`. Mova o check pra cima:
```js
if (dropIdx === srcIdx) return null;
var d = STATE.grid[dropIdx];
```

**4. BUG GRAVE — "melhores candidatas" é *first-match*, não "melhor".** O plano diz "2 melhores candidatas" mas o código pega o **primeiro adjacente na ordem do array `GRID_ADJ`**. Isso é **não-determinístico do ponto de vista do jogador**: com 4 Tier-1s, sempre sobra a mesma por azar de ordem de adjacência, não por critério visual.
   - **Sugestão**: priorize o partner **mais próximo do `srcIdx`** (Manhattan), assim o cluster "parece" visualmente coeso e o sobrevivente fica na posição mais distante — o que combina com o pulse preview.

**5. Edge case silencioso — cluster de 3 onde `srcIdx` é adjacente a `dropIdx` mas não a `partner`.** A regra do user diz "pelo menos 2 das 3 células geometricamente adjacentes". Seu código garante `partner↔dropIdx` adjacente, mas não valida `srcIdx` posicionalmente. Isso está correto (regex satisfeita: dropIdx↔partner = 1 par adjacente), mas **UX-wise** o jogador pode arrastar de longe e o cluster "teletransporta". Confirmar se é desejado.

## 🟡 Race conditions

**6. `pointermove` dispara updatePullPreview em cada evento (~120Hz em alguns devices).** `querySelectorAll('.grid-cell')` + 8 `classList.remove` + novo `querySelector` por célula = thrashing. **Use rAF coalescing** (sugestão do GPT-5.3 procede):
```js
var pullRafId = 0, pendingHover = -1;
function schedulePullPreview(src, hover) {
  pendingHover = hover;
  if (pullRafId) return;
  pullRafId = requestAnimationFrame(function(){
    pullRafId = 0; updatePullPreview(src, pendingHover);
  });
}
```

**7. Cache dos `.grid-cell` nodes em init** (`DRAG.cellNodes = [...]` indexado por `data-idx`). Elimina 9 queries/frame.

**8. Cleanup de `.pulling` em `pointerup` / `pointercancel` / blur.** Se o pointer sair por `pointercancel` (interrupção do SO, chamada, etc.) sem `pointerup`, classes `.pulling` ficam grudadas. Adicione handler em `pointercancel` + `visibilitychange`.

## 🟡 Ghost offset — `position: fixed` é a escolha certa

**9. ✅ Solução `position: fixed` + `clientX - 100` é robusta e correta.** `fixed` ancora no viewport ignorando `transform` de ancestrais — exatamente o caso aqui (`#game` tem `transform: scale`).
   - ⚠️ **Caveat conhecido**: `transform` em ancestral **normalmente** cria containing block para `fixed` (spec CSS), transformando `fixed` em `absolute`-like. Em Safari iOS especialmente. **Teste obrigatório antes de merge.** Se der problema, anexar ghost a `document.body` (fora de `#game`) resolve.
   - Remova `width/height: 200px` hardcoded — use `--ghost-size` var ou `offsetWidth/2` calculado, senão assets diferentes quebram.

**10. Scale do harness + `fixed`**: se `fixed` funcionar, o ghost **não** será escalado junto com `#game`. Isso é provavelmente desejado (segue o dedo 1:1), mas visualmente o ghost pode parecer "grande demais" vs células escaladas. Confirmar com mockup.

# SEGURANÇA — HeroMerge v0.3 (Papel: Claude Sonnet 4.6)

## Vetores de Ataque Identificados

### 1. CLIENT-SIDE TRUTH — Crítico
```
STATE.grid vive 100% no cliente. Com GameGuardian ou
Cheat Engine (via emulador), attacker edita diretamente:
  STATE.grid[x].tier = 5  // tier máximo sem merge
  STATE.mergesCompleted += 9999
```
**Nenhuma validação server-side** detectável no código enviado.

**Mitigação obrigatória:** Hash de integridade do grid enviado ao servidor a cada merge. Server recalcula se o merge é legítimo (tiers corretos, célula existia, transição válida). Firebase Rules devem rejeitar writes que não passem pela Cloud Function.

---

### 2. `computeMergeCluster` — Race de Estado
O cluster é calculado no `pointermove` mas consumido no `pointerup`. Entre esses dois eventos, **um segundo dedo pode alterar `STATE.grid`**:

```
Dedo A: calcula cluster [2, 3, 6] — todos tier-2
Dedo B: merge independente consome cell[3] → null
Dedo A: pointerup → tryMergeCells lê cluster [2, 3, 6]
         STATE.grid[3] é null agora → comportamento indefinido
```

**Verificação em `tryMergeCells`:** o código atual em B faz `null` sem revalidar o cluster. Precisa:
```js
// ANTES de nullar, re-checar cada cell do cluster
for (var i = 0; i < cluster.length; i++) {
  if (!STATE.grid[cluster[i]] ||
      STATE.grid[cluster[i]].tier !== expectedTier) return false;
}
```

---

### 3. Multi-touch — `updatePullPreview` Sem Guard
```js
// Attacker com dois dedos simultâneos:
// Touch 1: DRAG.srcIdx = 2, tier-2
// Touch 2: dispara segundo pointerdown enquanto Touch 1 ativo
//          DRAG.srcIdx sobrescrito = 5, tier-3
// Touch 1 sobe (pointerup): srcIdx=5, dstIdx=celula-tier-2
// → tryMergeCells com tiers diferentes → retorna false
// MAS updatePullPreview já marcou células erradas com .pulling
```
**Mitigação:** Flag `DRAG.active` checada em todo handler. Segundo `pointerdown` ignorado se `DRAG.active === true`.

---

### 4. Drag de Tier 5 (Cap) — Ausente em `computeMergeCluster`
```js
// Proposta em A checa s.tier >= 5 → return null ✓
// MAS: .pulling ativa em células tier-5 via updatePullPreview
// se attacker forçar tier inconsistente (memory edit):
//   tier = 0 ou tier = -1 → tier + 1 = 0 ou -1 no grid
```
**Mitigação:** Validar `tier >= 1 && tier <= 5` em **toda** leitura de `STATE.grid`, não apenas no merge final.

---

### 5. `updatePullPreview` — Cleanup Falhando em `pointerup`
Se `pointerup` ocorrer **fora do grid** (dedo saiu da tela), o handler de `pointerup` pode não remover `.pulling`:
```js
// Cenário: drag rápido para fora da janela
// pointerup dispara em document, hoverIdx = -1
// updatePullPreview limpa → OK
// MAS: se setPointerCapture falhar em Safari iOS,
//      pointerup não é garantido no document
//      → células ficam em .pulling infinitamente
//      → animação CSS roda para sempre, drena bateria
```
**Mitigação:**
```js
function clearAllPullStates() {
  var cells = document.querySelectorAll('.grid-cell');
  for (var i = 0; i < cells.length; i++) {
    cells[i].classList.remove('pulling', 'match');
  }
}
// Chamar em: pointerup, pointercancel, visibilitychange
document.addEventListener('pointercancel', clearAllPullStates);
document.addEventListener('visibilitychange', function() {
  if (document.hidden) clearAllPullStates();
});
```

---

### 6. `position: fixed` + Scale do Harness — Mobile Safari
`#game` usa `transform: scale(...)`. Em Safari iOS < 15, `position: fixed` dentro de um ancestral com `transform` **vira `position: absolute` em relação ao ancestral transformado**, não ao viewport. O fix proposto (mover para `position: fixed`) **quebra em Safari iOS**.

**Mitigação:** Manter ghost como filho de `document.body` diretamente:
```js
// No init:
document.body.appendChild(document.getElementById('drag-ghost'));
// ghost.style.position = 'fixed' → seguro fora de qualquer transform
```

---

### 7. `tutorialOnFirstMerge` — Race com Flash
Se `spawnGridFlash` dispara animação CSS e em seguida `tutorialOnFirstMerge` reposiciona elementos DOM, o reflow pode **cancelar a animação** antes de completar, ou pior, o tutorial pode avançar estado antes do merge ser persistido.

**Mitigação:** Tutorial avança somente em callback de animação ou em `setTimeout` mínimo após confirmar `STATE.mergesCompleted` incrementou.

## Resumo de Prioridades

| Vetor | Severidade | Client-only? |
|---|---|---|
| STATE.grid sem validação server | **Crítica** | Sim |
| Multi-touch race em srcIdx | Alta | Sim |
| Tier inválido (memory edit) | Alta | Sim |
| pulling não limpa em cancel | Média | Sim |
| Safari fixed+transform | Média | Não |
| Tutorial race com flash | Baixa | Não |

**Regra geral:** qualquer valor que incrementa recurso (tier, mergesCompleted, gold) deve ser calculado e validado server-side. O cliente propõe; o servidor decide.

2) Casos onde falha  
- Contraexemplo geométrico: src=0, drop=3, partner=2 (mesmo tier). Merge passa no código, mas só 3-2 adjacente (1 aresta).  
- Ambiguidade com 4+ peças: drop tem dois parceiros válidos; preview pode piscar entre opções se critério não for determinístico/estável.  
- TOCTOU: preview mostra cluster A, no `pointerup` resolve cluster B se grid mudou (multi-touch/SUMMON paralelo/tutorial).  
- Conflito CSS: `.match` e `.pulling` ambos animam `box-shadow`; `.pulling` também usa `transform: scale` no próprio `.grid-cell`, podendo alterar hit-test/visual “pula célula”.  
- Cleanup incompleto: se `pointercancel`, `lostpointercapture` ou drop fora do grid, classe `.pulling` pode ficar presa.  

3) Correção proposta  
- Definir regra formal do trio: `{src,drop,partner}` com pelo menos 2 adjacências entre pares. Implementar score determinístico por parceiro:  
  `score = adj(drop,partner)*2 + adj(src,drop) + adj(src,partner)`; escolher maior score; empate por menor índice.  
- Unificar preview+commit: mesma função pura retorna `cluster + score`; no `pointerup` usar snapshot do grid ou bloquear inputs concorrentes durante drag (lock curto).  
- Performance: cache de `.grid-cell` por índice; em move usar `requestAnimationFrame`; atualizar só diff entre cluster anterior/novo (não limpar tudo).  
- CSS: `.pulling` deve sobrescrever `.match` (ex.: `.grid-cell.pulling { animation: ... !important; }`) e animar pseudo-elemento interno, não a célula inteira.  
- Ghost robusto: anexar em `document.body`, `position: fixed`, `left/top = clientX/clientY`, manter `translate(-50%,-50%)`; limpar em `pointerup/cancel/lostpointercapture`.

  [1,4], [0,2,5], [1,3,6], [2,7],
  [0,5], [1,4,6], [2,5,7], [3,6]
];

var GRID_COLS = 4;
var LAST_PULL_KEY = '';

function manhattanIdx(a, b) {
  var ax = a % GRID_COLS, ay = Math.floor(a / GRID_COLS);
  var bx = b % GRID_COLS, by = Math.floor(b / GRID_COLS);
  return Math.abs(ax - bx) + Math.abs(ay - by);
}

function areAdjacent(a, b) {
  var list = GRID_ADJ[a];
  for (var i = 0; i < list.length; i++) {
    if (list[i] === b) return true;
  }
  return false;
}

function computeMergeCluster(srcIdx, dropIdx) {
  if (srcIdx < 0 || srcIdx > 7 || dropIdx < 0 || dropIdx > 7) return null;
  if (srcIdx === dropIdx) return null;

  var s = STATE.grid[srcIdx];
  var d = STATE.grid[dropIdx];
  if (!s || !d) return null;
  if (s.tier >= 5) return null;
  if (d.tier !== s.tier) return null;

  var adj = GRID_ADJ[dropIdx];
  var best = -1;
  var bestScore = 999999;

  for (var i = 0; i < adj.length; i++) {
    var idx = adj[i];
    if (idx === srcIdx || idx === dropIdx) continue;

    var u = STATE.grid[idx];
    if (!u || u.tier !== s.tier) continue;

    // Menor score vence:
    // - bônus se também adjacente ao src original, formando cluster mais "real"
    // - depois menor distância ao src
    // - depois menor índice
    var score = 0;
    if (!areAdjacent(idx, srcIdx)) score += 100;
    score += manhattanIdx(idx, srcIdx) * 10;
    score += idx;

    if (score < bestScore) {
      bestScore = score;
      best = idx;
    }
  }

  if (best < 0) return null;
  return [srcIdx, dropIdx, best];
}
