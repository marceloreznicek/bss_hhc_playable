- **Engine:** Unity 6, C# (no async/await in resolver вЂ” pure synchronous per-tick loop).
- **Pillar 2 вЂ” battles are deterministic.** Same `(party, dungeon, focus, floor)` MUST yield byte-identical resolver output every time. The resolver uses a single seeded `System.Random(seed)` for enemy targeting. Anything new must remain a pure function of state. NO `Time.deltaTime`, NO `UnityEngine.Random`, NO non-deterministic dictionary iteration, NO floating-point order-of-operations divergence.
- **Combat model:** Strict turn-cycling. Hero0 attacks в†’ ... в†’ HeroN-1 attacks в†’ Enemy0 attacks в†’ ... в†’ EnemyM-1 attacks в†’ end-of-round status tick в†’ repeat. Each "atomic action" = one hero attack OR one enemy attack OR end-of-round step.
- **Two combat code paths exist today:**
  1. `CombatResolver.Resolve(...)` вЂ” pure C#, deterministic, used by `RunController.PreviewNextFloor()` to project floor outcomes. Has full mana/cast/status wiring. Returns a `CombatResult` with a `List<TickEvent>` log including `TickEventKind.Cast` events.
  2. `UIController.RunLiveCombat(...)` вЂ” animation-driven coroutine that re-implements the same combat for the player to watch. **Currently does mana gain locally but does NOT call ResolvePendingCasts and does NOT apply cast effects.** This is the main thing this feature must fix.
- **CSV-driven tuning is mandatory.** Hardcoded constants for tunable values are not acceptable.
- **Roster:** seeded in code today (`MetaState.SeedDefaultRoster`). 5 heroes wired: Knight (id=1), Rogue (id=2), Mage (id=3), Cleric (id=4), Ranger (id=5). 9 more hero prefabs exist in `Assets/CharacterPrefabs/` but are not yet registered. **Design decision under review:** how to wire ability/mana profiles for the 5 wired heroes in a way that scales to all 14 (and beyond) without a code change.

**D1. RunLiveCombat: replicate cast logic vs consume resolver log.**
Two options:
- (A) Add inline `ResolvePendingCasts`-equivalent in `RunLiveCombat`, mirroring the resolver's logic. Pro: keeps the existing animation-driven structure. Con: now there are TWO copies of the cast logic; drift is inevitable.
- (B) Refactor `RunLiveCombat` to consume the deterministic resolver's `TickEvent` log: call `Resolve(...)` once at floor start, then animate the log entry-by-entry. Pro: single source of truth. Con: bigger change; must reconcile animation pacing with log-driven playback.

**The current implementation is closer to (A).** Mana gain is already inlined locally (UIController.cs lines 1654вЂ“1660 and 1738вЂ“1750). What's missing is the cast-resolution call.

**D2. Banner trigger: derived from local state vs resolver event.**
- (A) After every atomic action in `RunLiveCombat`, scan heroes for `mana >= maxMana` and trigger the banner inline. Apply the cast effect inline (mirroring `CombatResolver.CastAbility`).
- (B) Run the deterministic resolver in advance for this floor; iterate its `TickEvent` log; when a `TickEventKind.Cast` event is reached, pause animation and play the banner.

**D3. Hero ability data: where does it live?**
The user explicitly wants future-proof. Options:
- (A) Set fields directly in `MetaState.SeedDefaultRoster` per hero (e.g., add an extra arg to `BuildHero` or assign fields after construction). Adding a new hero = code edit.
- (B) New CSV `Balance/balance_heroes.csv` with columns `heroId,maxMana,startingMana,manaPerAttack,manaPerHit,manaPerHitCap,manaLockTicks,abilityId`. `MetaState` loads this after seeding the roster and overlays the values. Adding a new hero = CSV row only.
- (C) Add the ability columns to a single `balance_heroes.csv` that ALSO carries the base stats (heroName/baseHp/baseAtk/etc.). Replace the in-code `SeedDefaultRoster` entirely.
**Spec recommends (B) for v1** (smallest change that is still data-driven), with (C) as a future migration. **Open question: is (B) the right ceiling, or is (C) worth doing now since we're touching the file?**

**D4. Banner placement in scene hierarchy.**
The existing combat HUD lives on `combatPanel` (single Canvas, sortingOrder 0). Spec recommends adding `CastBannerRoot` as a sibling of `fightBanner` so default sibling order puts it on top. Risk: future modal dialogs may need their own canvas; banner may need its own canvas to avoid sibling-order conflicts.

**D5. Cast effect timing relative to banner.**
- (A) Apply the cast effect (HP delta, status add, mana reset) at banner-start. Pro: target HP bar updates while banner is up вЂ” visual coherence. Con: if combat is "supposed to end" on the cast, the end-of-fight check might fire while the banner is still up.
- (B) Apply at banner-end. Pro: banner is purely a "pre-show" curtain. Con: feels disconnected вЂ” the player watches the banner finish, then the heal lands.
**Spec recommends (A) (apply at banner-start).** The end-of-fight check is gated until the banner queue drains; the existing `RunLiveCombat` settle path handles this naturally.

**D6. Multi-cast queue blocking the resolver state.**
During banner queue drain (1s Г— N), the live combat is paused but the deterministic resolver state is locked at "post-N-casts." The next atomic action resumes from there. Concern: if the player force-closes the app mid-banner, no save state captures "we were 1 banner into a 3-banner queue." Spec accepts this вЂ” banner queue is in-memory transient.

public static class CombatResolver
{
    public static CombatResult Resolve(IList<HeroStats> heroes, IList<EnemyStats> enemies,
        int focusEnemyIndex, int seed, DungeonRule rule)
    {
        // ... seeds rng, builds parallel status lists, calls ResolvePendingCasts at floor start ...
        // ... main loop applies hero attacks в†’ mana gain в†’ ResolvePendingCasts ...
        // ... enemy attacks в†’ mana gain в†’ ResolvePendingCasts ...
        // ... end-of-round TickAndPruneStatuses + DecrementManaLocks + ResolvePendingCasts ...
    }

    static void ResolvePendingCasts(List<HeroStats> heroList, ..., List<TickEvent> log)
    {
        for (int hi = 0; hi < heroList.Count; hi++)
        {
            if (heroList[hi].hp <= 0) continue;
            if (IsInterrupted(hi)) continue;
            if (heroList[hi].manaLockTicksRemaining > 0) continue;
            if (heroList[hi].maxMana <= 0) continue;
            if (heroList[hi].mana < heroList[hi].maxMana) continue;

            CastAbility(hi, ...);
            var h = heroList[hi];
            h.mana = 0;
            if (h.manaLockTicks > 0) h.manaLockTicksRemaining = h.manaLockTicks;
            heroList[hi] = h;

            if (!AnyAliveEnemy(enemyList)) return;
            if (!AnyAliveHero(heroList)) return;
        }
    }

    static void CastAbility(int hi, ..., List<TickEvent> log)
    {
        var def = AbilityRegistry.Get(heroList[hi].abilityId);
        int magnitude = ComputeMagnitude(heroList[hi], def);

        tick++;
        log.Add(new TickEvent {
            tick = tick,
            kind = TickEventKind.Cast,
            actor = $"Hero{hi}",
            target = def.targetType.ToString(),
            damage = magnitude,
            actorHpAfter = heroList[hi].hp,
            targetHpAfter = 0
        });

        if (def.effectType == AbilityEffectType.None || magnitude == 0) return;

        switch (def.targetType) {
            case AbilityTargetType.SelfOnly:        ApplyEffectToHero(hi, hi, def, magnitude, ...); break;
            case AbilityTargetType.FocusedEnemy:    /* fallback to FirstAliveEnemy */ break;
            case AbilityTargetType.FirstAliveEnemy: /* ... */ break;
            case AbilityTargetType.FirstAliveAlly:  /* fallback to SelfOnly */ break;
            case AbilityTargetType.LowestHpAliveAlly: /* fallback to SelfOnly */ break;
            case AbilityTargetType.AllEnemies:      /* iterate enemies */ break;
            case AbilityTargetType.AllAllies:       /* iterate allies */ break;
        }
    }

    static int ComputeMagnitude(HeroStats caster, AbilityDef def)
    {
        int basis = def.scale == AbilityScale.CasterMaxHp ? caster.maxHp : caster.atk;
        long m = (long)basis * def.magnitudeCoeffMilli / 1000L;     // fixed-point milli to avoid float
        if (m < 0) m = 0;
        if (m > int.MaxValue) m = int.MaxValue;
        return (int)m;
    }
}

            int rawDmg = Math.Max(1, enemies[ei].atk - heroes[targetIdx].def);
            int dmg = Math.Max(1, RoundToInt(rawDmg * dmgMult));
            var h = heroes[targetIdx]; h.hp = Math.Max(0, h.hp - dmg); heroes[targetIdx] = h;

            // Mana on hit вЂ” already wired (lines 1738-1750)
            if (heroes[targetIdx].maxMana > 0) {
                var hHit = heroes[targetIdx];
                int manaGain = hHit.manaPerHit + rawDmg;
                if (hHit.manaPerHitCap > 0 && manaGain > hHit.manaPerHitCap) manaGain = hHit.manaPerHitCap;
                if (manaGain > 0) {
                    hHit.mana = Math.Min(hHit.maxMana, hHit.mana + manaGain);
                    heroes[targetIdx] = hHit;
                }
            }

            // ... animation: enemy attacks, hero HP/mana bar update ...
            yield return PlayAttack(...);
            // <-- MISSING: ResolvePendingCasts call HERE
        }

        // <-- MISSING: TickAndPruneStatuses + DecrementManaLocks + ResolvePendingCasts at end of round
    }

    // Settle outcome
    var hpAfter = new int[heroes.Count];
    for (int i = 0; i < heroes.Count; i++) hpAfter[i] = heroes[i].hp;
    var outcome = AnyAliveHero(heroes) ? FloorOutcome.Cleared : FloorOutcome.Wiped;
    run.ApplyFloorOutcome(hpAfter, outcome);
    // ... post-fight UI updates ...
    isPlayingFloor = false;
}
