# Revisão de Segurança — Character Special Abilities v1

## 1. Vetores de Ataque Críticos

### 1.1 Cliente como fonte de verdade para mana/status
**Risco alto.** O spec inteiro é server-side? Se o resolver roda APENAS no cliente (Unity), um atacante com GameGuardian pode:
- Editar `hero[hi].mana` diretamente em memória → forçar cast instantâneo
- Editar `ticksRemaining` de status → tornar Shield permanente
- Editar `hp` de inimigos → pular floors

**Mitigation:** Se houver qualquer placar competitivo (PvP/torneio), o seed + inputs devem ser enviados ao servidor e o resultado reprocessado server-side. Para PvE solo, documentar explicitamente que o modelo de ameaça aceita cheating local.

### 1.2 Struct-List Aliasing — Vetor de State Corruption
**Item 3 do prompt, mas com implicação de segurança:** `HeroStats` como struct com `List<ActiveStatus>` cria aliasing de referência. Em modo multiplayer ou replay, um state clone corrompido pode produzir resultados divergentes do servidor — abrindo brecha para replay fraudulento onde o cliente envia resultado manipulado.

**Mitigation:** `ActiveStatus` lista deve ser deep-copied em toda cópia de `HeroStats`. Garantir via construtor explícito.

### 1.3 Overflow de Recursos via manaPerHitCap = 0
Se CSV carregar `manaPerHitCap = 0`, a expressão `min(rawDamage, 0)` resulta em 0 — comportamento silencioso. Se `maxMana = 0`, `mana >= maxMana` é sempre verdadeiro → cast infinito por tick.

**Mitigation obrigatória:** Validação no loader: `maxMana <= 0` → warn + force `maxMana = 1`, desabilitar ability.

### 1.4 Save File — Status Effects Persistidos
Spec item 8 determina clear de statuses no floor start, mas se o save file serializar `activeStatuses` e for modificado externamente (GameGuardian file editor), o hero pode entrar no floor com Shield permanente.

**Mitigation:** Ao deserializar save, forçar `activeStatuses.Clear()` antes de `ResolvePendingCasts()`. Não confiar no estado serializado de statuses.

### 1.5 DamageOverTime — Pre-mitigation no spec é ambíguo
Spec item 3 define mana gain on hit como `rawDamage = enemyAtk` (pre-mitigation). Se DoT também usar `magnitude` sem mitigação aplicada sobre Shield, um atacante que manipula `sourceHeroIndex` no save pode inflar DoT de heróis inexistentes.

**Mitigation:** Validar `sourceHeroIndex` está em `[0, heroCount-1]` ao deserializar. Tratar index inválido como no-op + log.

## 2. Determinismo — Riscos Adicionais

- `List<ActiveStatus>` não é value type: iteração de substituição ("replace on re-cast") usando `List.Find` pode divergir se implementada com LINQ (order não garantida em edge cases de igualdade)
- `dmgMult` é `float` — `System.Math.Round(rawDmg * dmgMult)` pode divergir entre plataformas ARM vs x86 se `dmgMult != 1f`. Usar aritmética inteira: `(rawDmg * misFocusMultiplierNumerator) / misFocusDenominator`

## 3. Invariantes de Teste Críticos

```
// Deve falhar se aliasing de lista estiver presente
Assert(clone.activeStatuses != original.activeStatuses) // referência diferente

// Determinismo
Assert(Resolve(state, seed) == Resolve(state, seed)) // byte-identical

// maxMana=0 não causa loop infinito
Assert(ResolvePendingCasts com maxMana=0 termina em <= 1 iteração)
```

**Veredicto:** Não shipar em modo com scores competitivos sem reprocessamento server-side do resolver. Para PvE, priorizar fix do aliasing de struct antes de qualquer merge.

# Pre-Code Review: Character Special Abilities v1

## 1. Determinism Risks (Critical)

### 1.1 Struct-List Aliasing — **BLOCKER**
`HeroStats` is a `struct`. Adding `List<ActiveStatus> activeStatuses` creates shared reference semantics on copy:

```csharp
var h1 = heroList[0];        // copies struct, but List ref is shared
h1.activeStatuses.Add(...);  // mutates the SAME list as heroList[0]
```

This breaks determinism if any code path copies the struct without deep-cloning the list. The current resolver already does `var h = heroList[tIdx];` — if `ActiveStatus` list is added, this pattern silently corrupts.

**Fix required before coding:** Either make `HeroStats` a `class`, or add a `Clone()` helper that deep-copies the list, and enforce its use everywhere the struct is read/modified.

### 1.2 Floating-Point in Status Tick
Spec says `StatBuff`/`StatDebuff` magnitude scales off `atk` via a coefficient. If coefficient is `float` and applied each tick, order-of-operations divergence is possible across runs.

**Fix:** Store magnitude as `int` in `ActiveStatus`. Pre-compute at cast time.

### 1.3 `System.Random` Re-seeding
Current code creates one `Random(seed)` at start. If `ResolvePendingCasts` needs RNG (future: random-target abilities), it must reuse the same instance. Creating new `Random()` inside the loop risks same-tick seeds.

**Flag:** Ensure `rng` is threaded through every method that might need it, even if v1 doesn't use it.

## 2. Edge Cases the Spec Misses

### 2.1 Status Tick on Early-Exit Rounds
If the last hero kills all enemies during the hero-side, the loop breaks **before** the enemy-side and before `TickAndPruneStatuses()`. Do statuses still tick? The spec says "end of enemy-side step" — but there is no enemy-side step if enemies are dead.

**Recommendation:** Tick statuses at end of every **full round** (after enemy-side OR after hero-side if enemies wiped). Add explicit rule.

### 2.2 DamageOverTime Application Timing
DoT is listed as a status type but its damage application timing is unspecified. Does DoT damage apply:
- At the same time as the global tick-down?
- Before or after the mana-gain-from-damage check?
- Does DoT damage count as "taking damage" for `manaPerHit`?

**Recommendation:** DoT damage applies during `TickAndPruneStatuses()`, after all attacks, before mana lock decrement. DoT damage does NOT trigger `manaPerHit` (it's not an enemy attack).

### 2.3 Shield vs DoT Interaction
Spec is silent. Does Shield absorb DoT damage? If yes, Shield must be checked during DoT application, not just during direct attacks.

**Recommendation:** Shield absorbs all damage (including DoT). Document.

### 2.4 StatBuff Timing
If a hero casts `StatBuff` on themselves during their attack trigger, does the buffed `atk` apply to the **same** attack's damage, or only future attacks?

Given the flow (`PerformHeroAttack` → `mana += manaPerAttack` → `ResolvePendingCasts`), the buff lands **after** damage. This is correct but non-obvious. Document.

### 2.5 `LowestHpAliveAlly` Self-Target
If caster is the lowest-HP alive ally, should the ability target self (effectively `SelfOnly`) or skip? The fallback rule says "→ `SelfOnly` if caster is the only alive ally" — but doesn't cover "caster is lowest among multiple."

**Recommendation:** `LowestHpAliveAlly` always picks the lowest-HP alive ally (could be self). Only falls back to `SelfOnly` when caster is the *only* alive ally.

### 2.6 Focus Clear on Target Death
Spec says `RunController.FocusEnemyIndex` resets on `ApplyFloorOutcome`, but mid-floor focus when the focused enemy dies is unhandled. The resolver's `PickHeroTarget` currently returns `-1` if no valid target — but with focus set to a dead enemy, what happens?

**Recommendation:** `PickHeroTarget` should auto-fallback to `FirstAliveEnemy` when focused enemy is dead, and `focusEnemyIndex` should be cleared to `-1` in the resolver state (not just at floor end).

### 2.7 Floor-Start Cast on Fresh Enemies
`ResolvePendingCasts()` runs after `startingMana` assignment and after enemies are spawned. If a hero has `startingMana >= maxMana` and ability is `AllEnemies`, the cast fires at floor start. Is this intended? It effectively gives a free pre-combat nuke.

**Recommendation:** Confirm with design. If unintended, move `ResolvePendingCasts()` to **before** enemy spawn, or add a `canCastAtFloorStart` flag.

### 2.8 Re-entrancy Safety
`ResolvePendingCasts` iterates heroes; a cast could (in future) apply mana to allies, pushing them to max and triggering another cast in the same loop iteration. The current sequential walk would handle this (the later hero would be checked after the buff lands). But if a cast could buff an **earlier** hero in the list, that hero would not get re-checked.

**Flag:** Document that v1 abilities cannot grant mana. If future abilities do, the loop must be re-entrant (repeat until no new casts).

## 3. C# / Unity Gotchas

### 3.1 Struct Mutation Pattern
Current code:
```csharp

h.hp -= dmg;
heroList[tIdx] = h;
```
With `List<ActiveStatus>` added, this pattern becomes dangerous because `h.activeStatuses` is a reference to the list inside `heroList[tIdx]`. Mutating `h.activeStatuses` **does** mutate the original (since it's the same list object). But **reassigning** `h.activeStatuses = newList` would not propagate back.

This is a subtle inconsistency: some mutations persist, others don't. **Strongly recommend converting `HeroStats` to a `class`.**

If `HeroStats` gains a `List<>` field, this copy shares list references. Any mutation inside `Resolve` (which is also used by `PreviewNextFloor`) would corrupt the live `Party` state.

```csharp
    public HeroStats CloneRuntime() {

        c.activeStatuses.Clear();
        c.activeStatuses.AddRange(activeStatuses);
```

3. **Status effects estão subespecificados**
   - Definir ordem única no fim do round: aplicar DoT/heal/status effects primeiro, depois decrementar/prunar, ou vice-versa. “tick-down” não diz quando DoT causa dano.
   - Definir se status ticka quando o combate acaba no meio do round. Sugestão determinística: só tickar após completar enemy-side loop e ainda houver ambos os lados vivos.
   - Definir Shield vs DoT: absorve ou não. Sugestão: Shield absorve todo dano recebido, inclusive DoT, para uma regra simples.
   - StatBuff/Debuff aplicado mid-round deve afetar imediatamente ações posteriores do mesmo round; mais simples e deterministicamente observável.

4. **ResolvePendingCasts: cuidado com reentrância e cascata**
   - Não chamar recursivamente. Um cast não deve disparar outro cast até o loop atual continuar em slot order.
   - Se efeitos futuros concederem mana, o herói anterior no slot não deve ser reconsiderado até o próximo boundary. Documentar como invariant.
   - Unknown ability deve consumir mana mesmo sem efeito.

5. **Targeting**
   - `FocusedEnemy` deve validar alive; se morto, cair para first alive, sem alterar `RunController.FocusEnemyIndex` durante resolver.
   - `LowestHpAliveAlly`: se caster é menor HP, target self. Fallback para self só quando não há outro vivo é redundante, mas ok.
   - AoE deve iterar por índice crescente; nunca por dictionary/set.

6. **CSV hardening**
   - `maxMana <= 0` é perigoso: todo boundary castaria. Definir default mínimo `1` ou tratar como invalid row e usar no-op ability com `maxMana=999999`.
   - Clamp `startingMana`, `manaPerAttack`, `manaPerHitCap` para `>=0`; rejeitar/normalizar negativos.
   - `abilityId` vazio → `None`, warning único por id/hero, não por cast.
   - Não usar `Dictionary.Values` para resolver ordem de habilidades/status. Lookup por id ok; aplicação sempre por listas ordenadas.

7. **Preview/performance**
   - Adicionar `CombatOptions { bool collectLog }`; preview pode não coletar log detalhado.
   - Evitar alocações por cast: targets via loops diretos, não `LINQ`, não listas temporárias.
   - Deep clone só no início do resolver, não por tick.
   - Status list capacity pequena inicial, exemplo `new List<ActiveStatus>(4)`.

8. **API mínima**
   - Estender resultado:
```csharp
public struct CombatResult {
    public FloorOutcome outcome;
    public int[] heroHpAfter;
    public int[] heroManaAfter;
    public int totalTicks;
    public List<TickEvent> log;
}
```
   - Se UI quer trajetória, adicionar opcionalmente mana em `TickEvent`: `actorManaAfter`, `targetManaAfter`, ou novo `AbilityEvent`.

9. **Testes obrigatórios**
   - Mesmo input resolve 100 vezes com hash byte-idêntico de outcome/hp/mana/ticks/log.
   - Preview não altera `RunController.Party` hp/mana/status.
   - Herói morto por hit ainda ganha mana, mas não casta.
   - Dois heróis cheios castam em slot order.
   - Unknown ability consome mana e não crasha.
   - Focus morto cai para first alive.
   - Status recast substitui, não duplica.

**RED TEAM REVIEW – DiverDungeon Character Special Abilities v1 (pre-code)**

- **StructListAliasingExploit**: Adicionar `List<ActiveStatus> activeStatuses` diretamente em `HeroStats` (struct). Execute: `var copy = heroList[0]; copy.activeStatuses.Add(newStatus);` — a lista é compartilhada por referência entre todas as cópias feitas em `Resolve()`, `PreviewNextFloor()` e clones. Impacto: preview altera estado do Party real, multi-device save/load divergem, preview floor X modifica mana/status de floor X-1 de forma não-determinística.

- **ResolvePendingCastsReentrancyViaStatBuff**: Embora v1 não tenha mana-gain-on-cast, o framework permite `StatBuff` que aumente `atk` ou `maxMana` indiretamente. Execute: cast de herói 0 aplica StatBuff que faz herói 1 atingir `mana >= maxMana` dentro do mesmo `ResolvePendingCasts()`. Impacto: se o loop continuar sem cópia de lista ou snapshot, itera sobre coleção modificada (InvalidOperationException ou skips silenciosos), quebra ordem de slot e determinism (resultado varia conforme ordem de aplicação).

- **ManaLockTickRaceWithEndOfRound**: `manaLockTicksRemaining` decrementado só em `TickAndPruneStatuses()` no final do round inimigo. Execute: cast que aplica manaLock=2 no último atomic action da rodada; o decremento ocorre antes ou depois do próximo `ResolvePendingCasts()` dependendo se o round terminou cedo por wipe. Impacto: herói pode castar 1 tick antes do esperado em runs vs preview, violando Pillar 2.

- **FloorStartCastOnFreshEnemiesDesync**: `on_floor_start` limpa statuses, seta `startingMana`, chama `ResolvePendingCasts()`. Execute: herói com `startingMana >= maxMana` e `abilityId` = AllEnemies/FirstAliveEnemy. Impacto: jogador vê dano em inimigos antes do primeiro turno visual (UI mostra inimigos full HP mas log tem dano), preview e real run divergem em totalTicks se o cast matar inimigos instantaneamente.

Como struct copia referência da List<ActiveStatus>, qualquer `Add()` ou `Clear()` em preview contamina o estado real do RunController. Execute: preview roda 60fps enquanto jogador abre menu. Impacto: mana inflacionado em runs com tide rule, preview (que recria rule) pode usar float rounding diferente se `(int)Round` for chamado em ordem errada, quebra determinism em seeds idênticos.

- **MultipleCastsPerTriggerOrderDependency**: `ResolvePendingCasts()` anda de slot 0 a N. Execute: herói 0 cast AllEnemies que aplica DoT em todos; herói 1 (agora com mana cheia por algum buff) casta Shield. Ordem fixa é boa, mas se alguém mover o loop para `while any at max` sem ordem de slot, ordem de cast vira não-determinística. Impacto: StatBuff/Debuff aplicados em sequência diferente alteram dano posterior no mesmo round.
