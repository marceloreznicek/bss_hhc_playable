### CombatResolver.UpsertStatus (replace-not-stack rule)
```csharp
static void UpsertStatus(List<ActiveStatus> bag, ActiveStatus s) {
    for (int i = 0; i < bag.Count; i++) {
        if (bag[i].type == s.type) { bag[i] = s; return; }
    }
    bag.Add(s);
}
```
Garante no máximo 1 entry por StatusType por target.

PONTOS A REVISAR
══════════════════════════════════════════════════════════════════

**§1. State propagation**
- `heroStatuses` é local em `RunLiveCombat`. Para `RefreshAllCombatBars` atualizar ícones, ele precisa receber a lista. Opções:
  - (A) Adicionar parâmetro: `RefreshAllCombatBars(heroes, enemies, heroStatuses)`. Mudança em todos call sites.
  - (B) Promover `heroStatuses` para field do `UIController`, populado em RunLiveCombat. Risco: estado vaza entre floors se não limpar.
  - (C) Ter um `UpdateHeroStatusIconsAll(heroStatuses)` separado, chamado pelos mesmos pontos que `RefreshAllCombatBars` (cast banner + end-of-round).
- Recomendar opção. Por que?

**§2. Sprite import + .meta files**
- O .meta precisa de GUID novo por arquivo. Posso gerar via `python3 -c "import uuid; print(uuid.uuid4().hex)"` ou similar.
- Risk: Unity pode regenerar .meta se detectar conflito. Aceitável?
- Cache estático vs cache no UIController.Awake — qual prefere?
- Deve fazer load ASAP (Awake) para que BuildCombat possa atribuir? Sim, BuildCombat é chamado em Awake.

**§3. Layout / posicionamento**
- HP bar é largura total da hero rect (anchoredPosition Y=-8, pivot Y=1). Significa que ela vai do Y=-8 para Y=-34 aproximadamente (sizeDelta height 26).
- Icons "bottom-right OVER the HP bar" → anchor (1, 0) na hero rt, sizeDelta 36×36, anchoredPosition (-4, -8) ou parecido.
- Shield icon à esquerda do dot icon? Ou empilhados verticalmente? User não especificou. Recomendar.
- Tamanho 36×36 está ok ou muito pequeno em 1080p?

**§4. Edge cases**
- Hero KO com status ativo: ícone deve sumir (status bag não é limpo no KO mas o herói está morto). Ou continuar mostrando? Recomendar "esconder se hp <= 0".
- Status com magnitude 0: pode acontecer? Shield drenado deveria ser pruned por AbsorbWithShield. DoT 0-magnitude indica dado errado.
- Shield magnitude diminui a cada hit (drena). Ícone precisa atualizar a cada enemy attack, não só cast. Mas o número só aparece em RefreshAllCombatBars que é chamado entre ações atômicas? Verificar. **Importante**: cada vez que um inimigo bate em um herói com shield, o shield drena. Preciso chamar UpdateHeroStatusIcons também ali.
- DoT damage tick em end-of-round drena duração mas não magnitude (DoT mantém magnitude até prune). Após prune (ticksRemaining=0), magnitude=N mas removido do bag. UpdateHeroStatusIcons após end-of-round mostraria sumir.

**§5. Performance**
- Update por hero por action atômica = barato.
- Texture loading: 2 sprites de 1024×1024 RGBA = ~8MB cada. Talvez import com max 256×256 no .meta. Recomendar.

**§6. Determinismo**
- UI puro, presentation-only. Não afeta resolver. Confirmar.

**§7. Save/load impact**
- Status icons são UI runtime; nenhum impacto em PlayerPrefs save. Confirmar.
