# Pre-Code Review: Cast Banner + 5-Hero Ability Activation

## §1. Determinism risks

**Good news:** `ResolvePendingCasts` in `CombatResolver.cs` is pure state — no `rng.Next()` calls inside it. The enemy-targeting rng remains synchronized with `RunLiveCombat`'s `System.Random(floor*1000 + heroes.Count)`. Confirmed safe.

**Risks to flag:**
- `HeroStats` is a **struct**. In `RunLiveCombat`, the existing pattern `var h = heroes[i]; h.mana = ...; heroes[i] = h;` is correct. When you add inline cast handling, ensure you re-fetch after `heroes[i] = ...` assignments — any closure over `heroes[hi]` inside a coroutine that yields will capture a stale copy. Audit every `yield return` between mana write and cast check.
- `AbilityRegistry.Get(abilityId)` — confirm it uses an ordered lookup (Dictionary is fine for single-key `Get`, but any iteration over its values must be avoided in the cast path).
- `ComputeMagnitude` uses `long` fixed-point (good). No floats in cast math. ✓
- **One latent bug:** `RunLiveCombat` does NOT currently call `ResolvePendingCasts` on floor-start. Resolver DOES (honors `startingMana >= maxMana`). This is a divergence vector today — add it.

## §2. RunLiveCombat structural change

**Vote: D1.A (inline), with one caveat.** D1.B (consume log) is architecturally cleaner but requires reconciling animation pacing with log entries that have no inherent time (the resolver emits events, not durations). You'd end up inventing a second playback layer. Given the resolver is locked and small, duplicating ~40 lines of cast orchestration is lower risk.

**Caveat / mitigation for drift:** extract the cast-resolution body into a shared static helper `CombatResolver.TryCastForHero(int hi, ref List<HeroStats>, ref List<EnemyStats>, ...)` that BOTH paths call. That kills the two-copies problem without restructuring `RunLiveCombat`. Strongly recommend doing this — it's ~15 lines of refactor and eliminates the principal D1.A downside.

**§2 call sites in `RunLiveCombat` (UIController.cs ~1607–1808):**
1. After floor-start mana reset (~line 1618) — add `ResolvePendingCasts`.
2. After hero attack mana gain (after line 1660, before `yield return PlayAttack`).
3. After enemy attack mana gain (after line 1750, before `yield return PlayAttack`).
4. End-of-round — **currently absent entirely**.

**§2 — status-tick gap (critical):** `RunLiveCombat` has NO end-of-round status processing. The resolver does (`TickAndPruneStatuses` + `DecrementManaLocks`). Without this in the live path:
- `bulwark` shield never decays on screen → Knight appears invincible for 10 ticks but resolver says shield is gone.
- `hemorrhage` DoT ticks never apply live → enemy HP bars freeze while resolver drains them.
- `mark_for_death` DEF debuff never expires live.

**This is a BLOCKER for 3 of the 5 abilities shipping in this feature.** `arcane_bolt` (burst) and `mending_light` (burst) work without status ticks; the other three don't. Recommendation: scope in end-of-round status processing for THIS feature. Flag to the user — the spec undershot.

## §3. Banner UI

- `WaitForSecondsRealtime(1.0f)` ✓ correct.
- **Input gating:** `raycastTarget=false` on banner children doesn't block `combatPanel`. Set a bool `isBannerPlaying` and gate focus/descend button handlers on it, OR add a transparent full-screen `GraphicRaycaster`-absorbing panel under the banner.
- Portrait wiring: `MetaState.Instance.GetPortrait(run.PartyData[casterIdx].id)` — but `run.Party` is `List<HeroStats>` (no `id` field on struct). You need a parallel `PartyIds` or `PartyData` list. Confirm this exists; if not, add it when plumbing party. **Flag this — easy to miss.**
- Fallback for null portrait: colored block using `HeroData.color`. Check `MetaState.Instance.GetHeroData(id)?.color`.

## §4. Hero ability data (D3)

**Vote: Option (B) for v1.** (C) is tempting but doubles the touch surface and risks breaking `SeedDefaultRoster` stability. (B) is the minimum change that satisfies "future-proof" for abilities specifically — adding hero #6 = one CSV row + one registry entry, zero code.

**However, register (C) as an explicit follow-up** in the spec — once we go to 14 heroes, having base stats in CSV will matter.

**Failure modes to enforce:**
- Missing CSV → log error, heroes get `maxMana=0` (no ability). Don't crash.
- `heroId` in CSV not in roster → warn, skip.
- Malformed row → war

# SECURITY REVIEW — Cast Banner + 5-Hero Ability Activation
**Role: Security / Anti-Cheat / Exploit Surface**
*DiverDungeon · Unity 6 · Mobile · Pre-Code*

---

## Threat Model Framing

This feature introduces two new attack surfaces: **(1) mana/cast state executed locally in `RunLiveCombat`** and **(2) hero ability data loaded from CSV at runtime**. Neither triggers a server call. The resolver is pure-local. That is the foundational risk this review addresses.

---

## VECTOR 1 — Client Is the Sole Source of Truth for Cast Outcomes

**Severity: HIGH (existing architecture risk, not introduced by this feature — but this feature widens it)**

`CombatResolver.Resolve()` and `RunLiveCombat` both run entirely on-device. Cast effects (Shield, DoT, Heal, Debuff) are applied locally. `run.ApplyFloorOutcome(hpAfter, outcome)` writes the post-floor HP state which feeds into subsequent floors, `PreviewNextFloor`, and ultimately progression gating.

**GameGuardian / IL2CppDumper attack:**
- Dump `HeroStats` struct layout → patch `mana` field to equal `maxMana` at any moment → hero casts every turn.
- Patch `HeroStats.manaPerAttack` or `manaPerHitCap` after roster load → infinite mana generation.
- Patch `AbilityDef.magnitudeCoeffMilli` (loaded from CSV into `AbilityRegistry` in memory) → `bulwark` shield = MaxHp × 10,000 instead of × 0.2.
- Freeze `enemies[*].hp` to 0 after cast → instant floor clear.

**New surface this feature adds:**
- `balance_heroes.csv` (D3 Option B) is loaded at runtime into `HeroData` fields. A rooted device can swap this file before launch, setting `manaPerAttack=9999` or `startingMana=maxMana` for all heroes → cast fires on turn 1 every floor.
- `balance_abilities.csv` already exists and is the same risk for `magnitudeCoeffMilli`.

**Mitigations:**
1. **Checksum both CSV files at load time.** Store an expected SHA-256 (or CRC32 is fine for casual tampering) baked into the build. Log a warning + revert to embedded defaults if mismatch. This does not stop a determined attacker with a recompiled APK, but stops file-swap cheating on unrooted devices.
2. **Embed hardcoded fallback defaults** for all 5 heroes' mana profiles in `MetaState.SeedDefaultRoster` so that if `balance_heroes.csv` fails integrity check, you silently fall back rather than crashing or accepting tampered values.
3. **Clamp all CSV-sourced values at load time** in `HeroData` (many already are in `ToStats()`): `manaPerAttack ≤ maxMana`, `magnitudeCoeffMilli ≤ reasonable_cap` (e.g., 10000 = 10×). Flag to designers if a row exceeds the cap — it's probably a typo anyway.

---

## VECTOR 2 — No Server Validation of Floor Outcomes

**Severity: MEDIUM (architectural, not new to this feature)**

`FloorOutcome.Cleared` / `FloorOutcome.Wiped` and the resulting `hpAfter` array are produced entirely client-side. A memory editor can:
- Write `FloorOutcome.Cleared` directly to the `run` object while a Wiped state is resolving.
- Edit `hpAfter[i]` to restore HP between floors (trivial with GameGuardian struct search).

**This feature does not fix this and does not make it worse**, but the addition of cast effects (especially Shield reducing incoming damage, Heal restoring HP) makes the floor outcome more variable and harder to validate heuristically server-side if server validation is ever added later.

**Mitigation for future-proofing:** design `run.ApplyFloorOutcome` to accept the full `CombatResult` (or a signed hash thereof) rather than raw `int[]`. This is a larger architectural change — flag it as tech debt for the server integration milestone, not a blocker for v1.

---

## VECTOR 3 — `balance_heroes.csv` (D3-B) Adds a New Unvalidated Code Path

**Severity: MEDIUM**

The proposed D3-B approach loads a CSV from `StreamingAssets/` or `Resources/` and overlays mana/ability fields onto `HeroData` objects after `SeedDefaultRoster` runs. Specific risks:

| Risk | Detail |
|------|---------|
| File not found | If CSV is absent, heroes silently have `maxMana=0` → no abilities. Not a security risk but a logic failure. |
| HeroId in CSV not in registry | Orphan rows are ignored. An attacker cannot inject a new hero via CSV alone (good). |
| `abilityId` in CSV not in `AbilityRegistry` | At cast time, `AbilityRegistry.Get(abilityId)` returns null → NullReferenceException in `CastAbility`. Need null-guard (the spec's "Unknown abilityId → displayName=???" path covers the UI but `CombatResolver.CastAbility` needs the same guard). |
| `startingMana > maxMana` | `ToStats()` already clamps. Confirm the CSV overlay path calls `ToStats()` or re-applies the same clamp — if the overlay writes directly to `HeroData` fields and `ToStats()` is called later, order matters. |

**Action:** Add a `ValidateHeroAbilityRow()` method called at CSV load time that logs and rejects any row where `manaPerAttack > maxMana` or `startingMana > maxMana`. Do not silently accept and clamp — make the designer-facing error loud.
